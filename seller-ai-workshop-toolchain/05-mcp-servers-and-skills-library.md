# MCP-серверы, коннекторы и библиотека скилов

**Навигация:** [← README](../README.md) | [← Методология](00-toolchain-methodology.md) | [← Воркшоп 1](workshop-01-data-api-storage.md)

## Зачем этот файл

Для курса важно разделить два слоя:

- **MCP-серверы / коннекторы** дают агенту инструменты: сходить в API, прочитать базу, открыть таблицу, скачать карточку, отправить сообщение.
- **Скилы** дают агенту методику: что делать по шагам, какие файлы читать, куда сохранять результат, как проверять вывод.

Простая формула:

```text
MCP / коннектор = что агент может сделать
скил = как агент должен выполнять задачу
```

## Базовая архитектура для продавца WB/Ozon

```text
Codex / Claude Code
  -> скилы и инструкции
  -> MCP-серверы / локальные коннекторы
  -> API WB/Ozon, Google Sheets, база, браузер, Telegram
  -> data/raw, data/processed, context, reports, tasks
```

На курсе не нужно сразу писать все серверы. Нужно показать карту: какие инструменты нужны и в какой последовательности их имеет смысл собирать.

## Какие MCP-серверы и коннекторы нужны

### 1. Marketplace API connector: данные из кабинета WB/Ozon

Назначение:

Получать официальные данные из API маркетплейса и складывать их в `data/raw`.

Инструменты:

- `get_products`;
- `get_sales_report`;
- `get_orders`;
- `get_stocks`;
- `get_supplies`;
- `get_ad_stats`;
- `get_finance_report`;
- `save_raw_response`.

Что получает:

- товары;
- продажи;
- заказы;
- остатки;
- поставки;
- рекламные расходы;
- финансовые отчеты.

Куда сохраняет:

```text
data/raw/wildberries/
data/raw/ozon/
logs/data-runs.md
```

Что важно:

- использовать официальные API и доступы;
- не хранить ключи в коде;
- писать дату, период и источник данных;
- не перезатирать старые выгрузки.

Это главный коннектор для первого воркшопа.

### 2. Card and competitor collector: карточки и конкуренты

Назначение:

Собирать информацию по карточкам товара, конкурентам, отзывам, вопросам, изображениям и ценам, если это разрешено правилами площадки и способом доступа.

Инструменты:

- `get_product_card`;
- `download_card_images`;
- `get_reviews`;
- `get_questions`;
- `get_competitor_cards`;
- `save_product_profile`;
- `save_competitor_snapshot`.

Что получает:

- текущую карточку товара;
- изображения;
- описание;
- характеристики;
- отзывы;
- вопросы;
- конкурентов;
- цены и офферы конкурентов.

Куда сохраняет:

```text
products/SKU/
  product-profile.md
  current-card.md
  images/
  reviews.md
  competitors.md
```

Что важно:

- не нарушать правила маркетплейсов;
- не копировать чужие фото, тексты и бренды;
- отделять сбор фактов от генерации выводов;
- сохранять дату снимка карточки, потому что карточки и цены меняются.

Это основной коннектор для финального воркшопа про карточки.

### 3. WB/Ozon knowledge connector: база знаний по правилам площадок

Назначение:

Давать агенту актуальный контекст по правилам маркетплейса, требованиям к карточкам, ограничениям, комиссиям, поставкам, рекламе и офертам.

Инструменты:

- `search_marketplace_docs`;
- `get_rule_by_topic`;
- `get_offer_terms`;
- `get_card_requirements`;
- `get_supply_requirements`;
- `save_rule_summary`.

Что хранит:

```text
knowledge/
  wildberries/
    rules-index.md
    card-requirements.md
    supply-rules.md
    advertising-rules.md
  ozon/
    rules-index.md
    card-requirements.md
    supply-rules.md
    advertising-rules.md
```

Зачем нужен:

AI не должен вспоминать правила по памяти. Он должен читать актуальную базу знаний или хотя бы локальные конспекты правил, которые регулярно обновляются.

Что важно:

- указывать дату обновления;
- хранить ссылку на источник;
- разделять "официальное правило" и "нашу рабочую интерпретацию";
- не делать финальные юридические выводы без проверки человеком.

### 4. Storage connector: база данных или локальное хранилище

Назначение:

Дать агенту доступ к накопленным данным без ручного поиска по папкам.

Варианты:

- файловое хранилище проекта;
- SQLite для старта;
- PostgreSQL для более серьезной системы;
- Google Sheets как промежуточный слой.

Инструменты:

- `list_datasets`;
- `get_latest_dataset`;
- `save_dataset`;
- `query_sales`;
- `query_stocks`;
- `query_reports`;
- `save_report_index`.

Куда пишет:

```text
data/
reports/
logs/
tasks/
```

Когда нужен:

- если выгрузок много;
- если нужно сравнивать периоды;
- если отчеты запускаются регулярно;
- если команда работает не только с файлами, но и с таблицами или базой.

### 5. Google Sheets connector

Назначение:

Сделать Google Sheets рабочим интерфейсом для команды.

Инструменты:

- `read_sheet`;
- `write_sheet`;
- `append_findings`;
- `update_status`;
- `sync_products`;
- `sync_weekly_report`.

Что удобно хранить:

- список товаров;
- себестоимость;
- статусы задач;
- недельные findings;
- план поставок;
- список карточек на доработку.

Что важно:

- Google Sheets не должен быть единственным источником правды для всего;
- нужно фиксировать, какие листы обновляет агент;
- опасные изменения лучше делать через отдельный лист `ai_draft`, а не сразу в боевую таблицу.

### 6. Telegram connector

Назначение:

Отправлять короткие сводки, напоминания и задачи собственнику или команде.

Инструменты:

- `send_owner_summary`;
- `send_manager_tasks`;
- `send_alert`;
- `send_report_link`.

Примеры:

- ежедневная сводка по остаткам;
- предупреждение о товаре без себестоимости;
- список карточек на проверку;
- ссылка на недельный отчет.

Что важно:

- Telegram не должен заменять отчет;
- в Telegram отправляем краткое резюме и ссылку на файл;
- не отправляем секреты, токены и полные финансовые выгрузки в открытый чат.

### 7. Media generation connector

Назначение:

Генерировать и редактировать фото/видео для карточек через API-провайдеров.

Провайдеры:

- OpenAI Images API;
- OpenRouter;
- fal.ai;
- Replicate;
- другие image/video API по необходимости.

Инструменты:

- `generate_product_image`;
- `edit_product_image`;
- `generate_lifestyle_scene`;
- `generate_video_concept`;
- `upscale_image`;
- `save_media_result`.

Куда сохраняет:

```text
products/SKU/media/
  prompts/
  outputs/
  selected/
  media-results.md
```

Что важно:

- проверять, что товар не искажен;
- не использовать чужие бренды и логотипы;
- не обещать свойства, которых нет у товара;
- финальный визуал должен проверять человек.

### 8. Browser / Playwright connector

Назначение:

Открывать страницы, делать скриншоты, проверять интерфейсы, смотреть, как выглядит страница или прототип.

Инструменты:

- `open_page`;
- `take_screenshot`;
- `inspect_element`;
- `download_file`;
- `check_visual_result`.

Где полезен:

- проверка лендинга;
- проверка Telegram-бота или формы;
- просмотр карточки;
- тест внутреннего мини-сервиса.

Что важно:

- не использовать для обхода правил площадок;
- для регулярного сбора данных лучше API, а не браузер;
- браузер полезен для проверки и демонстрации, но не всегда для промышленного сбора.

### 9. Documentation / knowledge MCP

Назначение:

Искать актуальную документацию по API, библиотекам, OpenAI, Anthropic, marketplace API и внутренним инструкциям.

Примеры:

- OpenAI Docs MCP;
- Context7;
- локальная база документации компании;
- база инструкций курса.

Инструменты:

- `search_docs`;
- `fetch_doc`;
- `summarize_doc`;
- `save_doc_summary`.

Зачем нужен:

Когда участник или агент пишет интеграцию, он должен опираться на документацию, а не на догадки.

### 10. Scheduler / automation connector

Назначение:

Запускать регулярные проверки.

Варианты:

- встроенные automations в Codex app;
- cron;
- GitHub Actions;
- локальный планировщик;
- отдельный сервер.

Инструменты:

- `run_daily_check`;
- `run_weekly_report`;
- `run_card_review`;
- `write_run_log`;
- `notify_result`.

Что важно:

- каждый запуск должен писать лог;
- отчеты не должны перезатираться;
- автозапуск не должен принимать финансовые решения без человека.

## Минимальный набор для курса

Для первой версии курса достаточно показать не все серверы, а 5 ключевых:

1. **Marketplace API connector**  
   Получение данных WB/Ozon.

2. **Storage connector**  
   Сохранение raw/processed/normalized, отчетов и логов.

3. **Google Sheets connector**  
   Интерфейс для команды.

4. **Card and competitor collector**  
   Карточки, отзывы, конкуренты.

5. **Media generation connector**  
   Фото и видео для карточек через API.

Остальные можно показать как следующий уровень:

- Telegram;
- Browser / Playwright;
- knowledge connector;
- scheduler;
- PostgreSQL;
- внутренний кабинет.

## Как это объяснять участникам

Нельзя начинать со слова "MCP" как с главной ценности. Для участника это звучит технически.

Лучше так:

```text
У агента есть рабочее место: файлы и папки.
Потом мы даем ему инструменты: API, таблицу, базу, браузер, Telegram.
MCP - один из способов подключить такие инструменты.
```

То есть:

```text
задача продавца
  -> данные
  -> инструмент доступа
  -> скил с методикой
  -> отчет или действие
```

## Библиотека скилов

Скилы нужны, чтобы участник не писал длинную инструкцию каждый раз заново.

Пример:

```text
$weekly-margin-check
Запусти еженедельную проверку маржи по последним данным.
```

Внутри скила уже описано:

- какие файлы читать;
- какие правила учитывать;
- какие данные считать устаревшими;
- куда сохранить отчет;
- как обновить индекс;
- что вывести собственнику.

## Какие скилы нужны для курса

### 1. `seller-data-intake`

Задача:

Принять новую выгрузку или API-ответ, описать структуру, сохранить raw-файл, предложить нормализацию.

### 2. `marketplace-api-map`

Задача:

Помочь участнику описать, какие endpoints WB/Ozon нужны под его задачи.

### 3. `data-normalizer`

Задача:

Перевести сырые CSV / Excel / JSON в понятный normalized-слой и Markdown-описание.

### 4. `memory-builder`

Задача:

Создать или обновить `context/`: товары, правила, себестоимость, оферты, поставки, источники правды.

### 5. `memory-auditor`

Задача:

Проверить, каких данных не хватает, что устарело, где конфликтуют источники.

### 6. `weekly-owner-report`

Задача:

Собрать недельный отчет для собственника: продажи, остатки, маржа, реклама, риски, задачи.

### 7. `stock-and-supply-check`

Задача:

Проверить остатки, поставки, риск out-of-stock и товары, где нужно действие.

### 8. `ad-spend-check`

Задача:

Проверить рекламные расходы, товары с плохой динамикой, кампании без понятного результата.

### 9. `product-card-audit`

Задача:

Проверить карточку: первый слайд, оффер, структура слайдов, отзывы, возражения.

### 10. `competitor-card-review`

Задача:

Собрать и сравнить карточки конкурентов, выделить идеи, которые можно адаптировать без копирования.

### 11. `media-brief-builder`

Задача:

На основе профиля товара и аудита собрать `media-brief.md` для фото/видео.

### 12. `media-prompt-pack`

Задача:

Подготовить промпты для OpenAI / OpenRouter / fal.ai / Replicate.

### 13. `designer-brief-builder`

Задача:

Собрать ТЗ дизайнеру или контент-менеджеру.

### 14. `telegram-summary`

Задача:

Сжать отчет в короткое сообщение для Telegram.

### 15. `integration-spec-builder`

Задача:

Собрать ТЗ на интеграцию, MCP-сервер, локальный скрипт или будущий плагин.

## Как устанавливать скилы в Codex

В Codex скил - это папка с файлом `SKILL.md`.

Минимальная структура:

```text
.agents/
  skills/
    weekly-owner-report/
      SKILL.md
```

Минимальный `SKILL.md`:

```md
---
name: weekly-owner-report
description: Запускается, когда нужно собрать еженедельный отчет по кабинету WB/Ozon: продажи, остатки, маржа, реклама, риски и задачи.
---

Ты собираешь еженедельный отчет для собственника.

Шаги:
1. Найди актуальные normalized-данные.
2. Прочитай context/source-of-truth.md.
3. Прочитай правила маржи и товаров.
4. Сформируй отчет в reports/weekly.
5. Обнови reports/index.md.
6. Добавь запись в logs/check-runs.md.
7. В конце дай список действий для команды.
```

Codex может использовать скил двумя способами:

- явно: пользователь пишет `$weekly-owner-report`;
- неявно: задача похожа на `description`, и Codex сам выбирает подходящий скил.

## Где хранить скилы

Для курса лучше разделить две зоны.

### 1. Библиотека в репозитории

Это публичная или учебная библиотека шаблонов:

```text
skills-library/
  seller-data-intake/
    SKILL.md
  weekly-owner-report/
    SKILL.md
  product-card-audit/
    SKILL.md
```

Эту папку можно коммитить в GitHub как учебный материал.

### 2. Рабочая папка Codex

Это место, откуда Codex реально подхватывает скилы:

```text
.agents/skills/
```

В этом проекте `.agents/` лучше не коммитить, потому что там могут появляться локальные настройки, эксперименты и приватные инструкции.

Поэтому схема такая:

```text
skills-library/
  -> учебные шаблоны в GitHub

.agents/skills/
  -> локально установленные рабочие скилы
```

## Как сделать библиотеку скилов

### Шаг 1. Создать папку `skills-library/`

```text
skills-library/
  README.md
  seller-data-intake/
    SKILL.md
  weekly-owner-report/
    SKILL.md
  product-card-audit/
    SKILL.md
```

### Шаг 2. Для каждого скила описать единый стандарт

В каждом `SKILL.md` должны быть:

- `name`;
- `description`;
- когда использовать;
- какие входные файлы нужны;
- какие MCP-инструменты или коннекторы нужны;
- какие файлы создавать;
- какие проверки сделать;
- чего не делать.

### Шаг 3. Добавить индекс библиотеки

Файл:

```text
skills-library/README.md
```

Содержание:

```text
# Библиотека скилов

| Скил | Когда использовать | Какие данные нужны | Что создает |
| --- | --- | --- | --- |
| seller-data-intake | новая выгрузка или API-ответ | data/raw | data summary |
| weekly-owner-report | недельный отчет | normalized + context | reports/weekly |
```

### Шаг 4. Установка скила локально

Вариант для занятия:

```text
Скопировать папку нужного скила из skills-library/ в .agents/skills/.
Перезапустить Codex, если скил не появился.
```

Пример:

```text
skills-library/weekly-owner-report/
  -> .agents/skills/weekly-owner-report/
```

### Шаг 5. Проверить вызов

Промпт:

```text
$weekly-owner-report
Собери отчет за последнюю неделю по демо-данным.
```

Если скил не вызывается явно, можно написать:

```text
Используй скил weekly-owner-report и собери отчет за последнюю неделю.
```

## Когда скил лучше, чем обычный промпт

Обычный промпт подходит, если задача разовая.

Скил нужен, если задача:

- повторяется;
- имеет одинаковые входные данные;
- должна сохранять результат в правильное место;
- должна обновлять индексы и логи;
- должна выполняться по правилам;
- будет использоваться участником, менеджером или командой много раз.

## Когда нужен плагин, а не просто скил

Скил описывает методику.

Плагин нужен, если вы хотите упаковать:

- несколько скилов;
- MCP-серверы;
- настройки;
- assets;
- инструкции установки;
- готовый набор инструментов для других пользователей.

Для курса логика такая:

```text
сначала делаем skills-library
  -> потом добавляем MCP-коннекторы
  -> потом упаковываем это в plugin
```

## Что показать на занятии

Минимальная демонстрация:

1. Показать, что MCP-коннектор дает агенту действие: получить данные, прочитать таблицу, сохранить отчет.
2. Показать, что скил дает агенту методику: как именно выполнить задачу.
3. Создать один учебный скил `weekly-owner-report`.
4. Скопировать его в `.agents/skills/`.
5. Вызвать `$weekly-owner-report`.
6. Показать, что результат сохранился в `reports/`, `reports/index.md` и `logs/check-runs.md`.
