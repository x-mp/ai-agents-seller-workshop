# Воркшоп 4: карточки, визуал, конкуренты и генерация медиа через API

**Навигация:** [← README](../README.md) | [← Воркшоп 3](workshop-03-cycles-skills-checks.md) | [Реклама и запуск](../ai-agents-seller-workshop-promo-and-next-steps.md)

## Суть

Этот воркшоп про финальный прикладной слой: как улучшать карточку товара с помощью данных, отзывов, конкурентов, AI-анализа и генерации медиа через API.

На этом этапе мы уже не разбираем MCP как отдельную тему. MCP, локальные инструменты и API-карты были в первом воркшопе. Здесь фокус другой: карточка, визуал, оффер, структура слайдов, фото, видео и ТЗ для обновления карточки.

Главный результат:

У участника появляется пакет для доработки одной карточки:

- профиль товара;
- аудит текущей карточки;
- карта конкурентов;
- боли и возражения из отзывов;
- новая структура слайдов;
- промпты для фото и видео;
- схема подключения API-провайдеров;
- папка с результатами генерации;
- ТЗ дизайнеру, контент-менеджеру или подрядчику.

## Что показываем

### 1. Профиль карточки товара

Начинаем не с картинки, а с продукта.

Цепочка:

```text
артикул / ссылка
  -> данные карточки
  -> описание
  -> характеристики
  -> изображения
  -> отзывы
  -> конкуренты
  -> product-profile.md
```

Что сохраняем:

```text
products/
  SKU123/
    product-profile.md
    current-card.md
    reviews.md
    competitors.md
    card-audit.md
    slide-structure.md
    designer-brief.md
    media-results.md
```

Промпт:

```text
Создай product-profile.md для товара.

Используй:
- описание карточки;
- характеристики;
- текущие изображения;
- отзывы;
- данные по конкурентам, если они есть.

Не придумывай свойства товара.
Если данных не хватает, выдели это отдельным списком.
```

### 2. Аудит текущей карточки

Проверяем не "красиво или некрасиво", а продающую ясность.

Что смотрим:

- понятно ли за 3 секунды, что это за товар;
- виден ли главный оффер;
- закрыты ли боли покупателя;
- есть ли доказательства;
- понятна ли комплектация;
- не перегружен ли первый слайд;
- какие возражения остаются после просмотра карточки.

Промпт:

```text
Проведи аудит карточки товара.

Оцени:
- первый слайд;
- структуру слайдов;
- описание;
- характеристики;
- отзывы;
- соответствие визуала реальному товару.

Не оценивай только дизайн.
Оцени, помогает ли карточка покупателю принять решение.

Сохрани вывод в products/SKU/card-audit.md.
```

### 3. Конкуренты и отзывы

Сравнение с конкурентами нужно не для копирования, а для понимания рынка.

Что собираем:

- какие офферы используют конкуренты;
- какие первые слайды повторяются;
- какие характеристики выносят в визуал;
- какие боли видны в отзывах;
- какие возражения нужно закрыть;
- где наша карточка слабее или сильнее.

Шаблон `competitors.md`:

```text
# Конкуренты

| Конкурент | Ссылка | Цена | Главный оффер | Сильный визуал | Что можно учесть |
| --- | --- | --- | --- | --- | --- |
```

Промпт:

```text
На основе отзывов и данных конкурентов собери карту улучшений карточки.

Раздели вывод:
- боли покупателей;
- возражения;
- сильные офферы конкурентов;
- что нельзя копировать;
- какие идеи можно адаптировать для нашей карточки.

Сохрани в products/SKU/competitor-insights.md.
```

### 4. Новая структура карточки

После аудита собираем не отдельную картинку, а структуру карточки.

Пример:

```text
Слайд 1: главный оффер и понятный вид товара
Слайд 2: ключевая выгода
Слайд 3: комплектация / размеры / состав
Слайд 4: сценарий использования
Слайд 5: сравнение или доказательства
Слайд 6: закрытие частого возражения
Слайд 7: детали, уход, гарантия или инструкция
```

Промпт:

```text
На основе product-profile.md, card-audit.md, reviews.md и competitors.md
создай новую структуру слайдов карточки.

Для каждого слайда укажи:
- цель слайда;
- главный текст;
- что должно быть на изображении;
- какую боль или возражение закрывает;
- какие данные нужно проверить перед производством.

Сохрани в products/SKU/slide-structure.md.
```

### 5. Генерация фото и видео для карточки через API

Здесь показываем, как перейти от идеи к медиа-пайплайну.

Важно: мы не обещаем, что AI заменит предметную съемку. Для карточек WB/Ozon важно соответствие реальному товару. Генерация полезна для концептов, фонов, lifestyle-сцен, инфографики, вариантов подачи, коротких видео и быстрых гипотез.

Базовая цепочка:

```text
product-profile.md
  -> media-brief.md
  -> image-prompts.md
  -> video-prompts.md
  -> provider request
  -> outputs/
  -> review
  -> selected/
  -> designer-brief.md
```

Папка:

```text
products/
  SKU123/
    media/
      briefs/
        media-brief.md
      prompts/
        image-prompts.md
        video-prompts.md
      outputs/
        openai/
        openrouter/
        fal/
        replicate/
      selected/
      provider-configs/
        openai-images.md
        openrouter.md
        fal.md
        replicate.md
```

Провайдеры, которые можно разобрать:

- OpenAI Images API для генерации и редактирования изображений;
- OpenRouter как единая точка для экспериментов с разными моделями, если нужная модель доступна у провайдера;
- fal.ai для быстрых генеративных image/video-моделей и batch-экспериментов;
- Replicate для запуска моделей через predictions API, webhooks и экспериментов с разными open-source или hosted-моделями.

Модели и направления, которые можно показать как кандидаты:

- OpenAI image generation / `gpt-image-*`;
- Grok image, если доступен через выбранного провайдера;
- Nano Banana / Nano Banana Pro, если доступен через fal.ai или другой провайдер;
- Kling и другие video-модели для коротких роликов;
- модели для product photography, background replacement, image editing, upscaling и video generation.

Перед живым запуском нужно проверить актуальную доступность моделей у провайдера. Названия и доступы меняются, поэтому в материале фиксируем не одну "вечную" модель, а архитектуру подключения.

Пример `.env.example`:

```text
OPENAI_API_KEY=your_key_here
OPENROUTER_API_KEY=your_key_here
FAL_KEY=your_key_here
REPLICATE_API_TOKEN=your_key_here
```

Промпт на `media-brief.md`:

```text
На основе product-profile.md и slide-structure.md создай media-brief.md.

Для каждого слайда опиши:
- цель изображения;
- что должно быть в кадре;
- какой текст нужен на слайде;
- какие свойства товара нельзя искажать;
- какие референсы нужны;
- какие варианты стоит сгенерировать.
```

Промпт на промпты для моделей:

```text
Создай набор промптов для генерации медиа.

Раздели:
- первый слайд;
- lifestyle-сцены;
- инфографика;
- детали товара;
- короткое видео 5-10 секунд.

Для каждого промпта укажи:
- провайдера-кандидата;
- модель-кандидат;
- входные изображения, если нужны;
- expected output;
- что проверить после генерации.
```

Промпт на подготовку API-запросов:

```text
На основе media-brief.md и image-prompts.md подготовь черновики API-запросов
для OpenAI Images API, OpenRouter, fal.ai и Replicate.

Не вставляй реальные ключи.
Используй переменные окружения из .env.example.
Сохрани конфиги в products/SKU/media/provider-configs/.
```

### 6. Проверка сгенерированных медиа

AI может сделать красивую картинку, которая плохо продает или искажает товар. Поэтому нужен слой проверки.

Что проверяем:

- товар похож на реальный;
- нет ложных свойств;
- нет запрещенных логотипов и чужих брендов;
- текст читается;
- слайд не перегружен;
- визуал соответствует правилам площадки;
- есть понятный оффер;
- картинка не вводит покупателя в заблуждение.

Промпт:

```text
Проверь сгенерированные изображения и видео для карточки.

Сравни их с product-profile.md и media-brief.md.

Для каждого варианта укажи:
- можно использовать;
- нужно доработать;
- нельзя использовать;
- почему;
- какие правки передать дизайнеру.

Сохрани вывод в products/SKU/media-results.md.
```

### 7. ТЗ дизайнеру, контент-менеджеру или подрядчику

Финальный результат не просто "картинки". Финальный результат - понятное ТЗ.

Шаблон:

```text
# ТЗ на доработку карточки

## Товар

## Цель обновления

## Что меняем

## Новая структура слайдов

## Референсы

## Сгенерированные варианты

## Что обязательно сохранить

## Что нельзя использовать

## Материалы

## Критерии приемки
```

## Что делают участники

1. Выбирают одну карточку.
2. Создают папку товара.
3. Собирают `product-profile.md`.
4. Делают аудит текущей карточки.
5. Добавляют конкурентов и отзывы.
6. Собирают новую структуру слайдов.
7. Создают `media-brief.md`.
8. Готовят промпты для фото и видео.
9. Проектируют подключение OpenAI / OpenRouter / fal.ai / Replicate.
10. Сохраняют результаты и делают ТЗ на доработку карточки.

## Артефакты

- `product-profile.md`;
- `card-audit.md`;
- `reviews.md`;
- `competitors.md`;
- `competitor-insights.md`;
- `slide-structure.md`;
- `media-brief.md`;
- `image-prompts.md`;
- `video-prompts.md`;
- `.env.example` для media-провайдеров;
- `provider-configs/`;
- `media-results.md`;
- `designer-brief.md`.

## Домашнее задание

1. Доработать пакет по одной карточке.
2. Проверить 3-5 конкурентов.
3. Сформировать 5-10 промптов для изображений.
4. Сформировать 1-2 промпта для короткого видео.
5. Проверить доступность выбранных моделей у провайдеров.
6. Подготовить ТЗ дизайнеру или контент-менеджеру.
7. Определить, какие элементы карточки можно поставить на регулярную проверку.
