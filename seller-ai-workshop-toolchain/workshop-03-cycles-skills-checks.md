# Воркшоп 3: циклы проверки, скилы и регулярные AI-процессы

**Навигация:** [← README](../README.md) | [← Воркшоп 2](workshop-02-memory-normalization.md) | [Далее: воркшоп 4](workshop-04-integrations-mcp-plugins-visuals.md)

## Суть

Этот воркшоп про то, как превратить разовые запросы к AI в повторяемые процессы: ежедневные, еженедельные и событийные проверки.

Главный результат:

У участника появляется 1-2 регулярных AI-процесса:

- что запускать;
- какие данные читать;
- какую память использовать;
- куда сохранять отчет;
- как не перезатирать прошлые результаты;
- как оформить это в скил или повторяемую инструкцию для агента.

## Что показываем

### 1. Что такое цикл проверки

Цикл:

```text
Получить данные
  -> прочитать контекст
  -> проверить по правилам
  -> записать выводы
  -> сохранить отчет
  -> обновить индекс и лог
  -> предложить действия
```

Примеры циклов:

- ежедневная проверка остатков;
- еженедельный отчет по марже;
- проверка товаров перед поставкой;
- проверка рекламных расходов;
- проверка карточек перед запуском рекламы;
- анализ отзывов раз в неделю;
- сводка задач менеджера.

### 2. Чем цикл отличается от разового промпта

Разовый промпт:

```text
Проанализируй продажи.
```

Цикл:

```text
Каждый понедельник:
1. взять последний sales_processed.csv;
2. прочитать cabinet-context.md и unit-economics-rules.md;
3. проверить SKU по правилам;
4. записать findings в reports/weekly;
5. обновить Google Sheet;
6. сформировать действия для собственника.
```

### 3. Файл регулярной проверки

Создаем:

```text
checks/weekly-margin-check.md
```

Шаблон:

```text
# Еженедельная проверка маржи

## Цель

## Какие файлы читать

## Какие правила применять

## Что проверить

## Формат результата

## Что записать в отчет

## Что считать проблемой

## Что не делать
```

### 4. Скил / инструкция для агента

Скил - это повторяемая инструкция для конкретного типа задачи. Он нужен, чтобы каждый раз не объяснять агенту одно и то же.

Важно объяснить человечески:

- в Claude Code и Codex это может называться и работать немного по-разному;
- смысл один: мы сохраняем методику задачи в отдельном файле;
- потом вызываем ее короткой командой или просим агента применить нужный скил;
- скил не заменяет данные, память и проверку человеком;
- скил фиксирует порядок действий, чтобы результат был повторяемым.

Примеры скилов:

- `wb-margin-check`;
- `product-card-review`;
- `review-analysis`;
- `supply-risk-check`;
- `weekly-owner-report`;
- `ad-spend-check`.

Пример структуры:

```text
skills/
  weekly-owner-report/
    SKILL.md
```

Пример `SKILL.md`:

```text
---
name: weekly-owner-report
description: Запускается для еженедельной проверки кабинета WB/Ozon по продажам, остаткам, марже, рекламе и задачам команды.
---

Перед началом:
- прочитай context/memory-index.md;
- прочитай context/source-of-truth.md;
- найди актуальные normalized-файлы;
- проверь reports/index.md, чтобы не перезаписать прошлый отчет.

Шаги:
1. Определи период проверки.
2. Найди входные данные.
3. Прочитай правила маржи и товаров.
4. Сформируй отчет в reports/weekly.
5. Добавь запись в reports/index.md.
6. Добавь запись в logs/check-runs.md.
7. В конце дай список действий для собственника и команды.
```

### 5. Регулярные задачи и накопление результатов

Показываем, что регулярная проверка не должна просто выводить ответ в чат.

Каждый запуск должен создавать след:

```text
checks/weekly-margin-check.md
  -> reports/weekly/2026-W27__weekly-margin-check.md
  -> reports/index.md
  -> logs/check-runs.md
  -> tasks/2026-W27__owner-actions.md
```

Что сохраняем:

- дату запуска;
- период данных;
- входные файлы;
- использованные правила;
- отчет;
- список действий;
- статус проверки: черновик, требует проверки, принято, передано в работу.

Шаблон `logs/check-runs.md`:

```text
# Лог регулярных проверок

| Дата запуска | Проверка | Период | Входные данные | Отчет | Статус |
| --- | --- | --- | --- | --- | --- |
| 2026-07-03 | weekly-margin-check | 2026-W27 | sales, costs, ads | reports/weekly/2026-W27__weekly-margin-check.md | требует проверки |
```

Промпт:

```text
Запусти регулярную проверку по инструкции checks/weekly-margin-check.md.

Перед запуском:
- проверь актуальные данные;
- прочитай context/source-of-truth.md;
- найди прошлые отчеты в reports/index.md.

После запуска:
- сохрани новый отчет с датой или номером недели;
- не перезаписывай старые отчеты;
- обнови reports/index.md;
- добавь запись в logs/check-runs.md;
- сформируй tasks/ с действиями для команды.
```

## Что делают участники

1. Выбирают одну регулярную проверку.
2. Описывают цикл.
3. Создают файл проверки в `checks/`.
4. Создают черновик скила в `skills/`.
5. Прогоняют тест на своих или демо-данных.
6. Сохраняют результат в `reports/`.
7. Обновляют `reports/index.md` и `logs/check-runs.md`.

## Практические циклы, которые можно выбрать

- еженедельная маржа;
- карточки для улучшения;
- поставки и остатки;
- отзывы и вопросы;
- рекламные расходы;
- ежедневная сводка менеджера.

## Артефакты

- папка `checks/`;
- файл первой регулярной проверки;
- папка `skills/`;
- черновик `SKILL.md`;
- первый отчет в `reports/`;
- `reports/index.md`;
- `logs/check-runs.md`;
- список задач в `tasks/`;
- понимание, как запускать цикл вручную и как потом автоматизировать.
