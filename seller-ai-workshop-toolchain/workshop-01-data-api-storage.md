# Воркшоп 1: получение данных, API, файлы и хранилище

**Навигация:** [← README](../README.md) | [← Методология](00-toolchain-methodology.md) | [Далее: воркшоп 2](workshop-02-memory-normalization.md)

## Суть

Это практический воркшоп про первый слой AI-инфраструктуры селлера: как получить данные и сохранить их так, чтобы дальше с ними мог работать агент.

Главный результат:

```text
У участника появляется структура проекта для данных и понимание, как подключать источники: API, CSV, Excel, Google Sheets.
```

## Что показываем

### 1. Карта источников данных селлера

Показываем, какие источники бывают:

- API маркетплейса;
- выгрузки из кабинета;
- рекламные отчеты;
- отчеты по продажам;
- остатки;
- поставки;
- карточки товаров;
- отзывы;
- Google Sheets;
- ручные файлы менеджера.

Практика:

Каждый участник составляет свою карту источников:

```text
Источник:
Какие данные дает:
Как часто нужны:
Как сейчас получаем:
Можно ли получить по API:
Где будем хранить:
Для каких проверок нужны:
```

### 2. Структура папки проекта

Показываем, что AI-агенту нужен порядок.

Пример:

```text
seller-ai-system/
  context/
    business-context.md
    products.md
    unit-economics-rules.md
  data/
    raw/
    processed/
    exports/
  reports/
    daily/
    weekly/
  prompts/
  skills/
  scripts/
  integrations/
  logs/
```

Практика:

Участник создает структуру под свой кабинет WB/Ozon.

### 3. Получение данных из CSV / Excel

Показываем не анализ, а путь:

```text
сырой файл -> проверка структуры -> нормализация -> сохранение processed-файла -> отчет для AI
```

Что делает агент:

- смотрит структуру файла;
- определяет столбцы;
- предлагает нормальные названия;
- проверяет пропуски;
- сохраняет описание данных в Markdown.

Промпт:

```text
Посмотри файл из папки data/raw.
Не анализируй бизнес-результаты.
Сначала опиши структуру данных: какие столбцы есть, что они означают, какие есть проблемы, каких полей не хватает.
Предложи, как привести файл к нормальной структуре для дальнейшего анализа.
```

### 4. Получение данных по API

Показываем методологию:

1. Найти документацию API.
2. Понять, какие endpoints нужны.
3. Создать тестовый запрос.
4. Сохранить ответ в `data/raw`.
5. Нормализовать ответ в CSV / JSON.
6. Записать описание данных в Markdown.
7. Не хранить ключи в коде.

Важно:

Реальные ключи на занятии не светим. Используем тестовые данные или демонстрационный ключ.

Структура файла `.env.example`:

```text
WB_API_KEY=your_key_here
OZON_CLIENT_ID=your_client_id_here
OZON_API_KEY=your_api_key_here
```

Что объясняем:

- API-ключ не пишется в промпт;
- ключ хранится в `.env`;
- `.env` не отправляется в чат и не коммитится;
- агенту можно дать `.env.example`, но не реальные секреты.

### 5. Google Sheets как простой интерфейс

Объясняем:

Google Sheets может быть первым "интерфейсом" системы:

- туда можно складывать очищенные данные;
- менеджер может проверять руками;
- собственник может смотреть результат;
- AI может готовить структуру таблицы;
- позже можно подключить API.

Практика:

Проектируем таблицы:

```text
products
sales_raw
sales_processed
ad_costs
supplies
weekly_report
ai_findings
```

## Что делают участники

1. Создают карту источников данных.
2. Создают структуру проекта.
3. Выбирают первый источник: CSV, Excel, Google Sheet или API.
4. Описывают, какие данные нужны и зачем.
5. Создают `data-inventory.md`.
6. Создают `.env.example`.
7. Пишут первую задачу агенту на проверку структуры данных.

## Артефакты

- структура проекта;
- `data-inventory.md`;
- `.env.example`;
- описание первого источника данных;
- схема Google Sheets;
- понимание, как данные будут попадать в систему.

## Домашнее задание

1. Подготовить 1 реальную выгрузку или тестовый API-ответ.
2. Положить в `data/raw`.
3. Попросить агента описать структуру.
4. Создать `data/processed` версию или план нормализации.
5. Не подключать реальные ключи без отдельной проверки безопасности.
