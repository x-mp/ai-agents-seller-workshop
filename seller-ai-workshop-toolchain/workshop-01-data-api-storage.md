# Воркшоп 1: интеграции, API, файлы и слой накопления данных

**Навигация:** [← README](../README.md) | [← Методология](00-toolchain-methodology.md) | [Далее: воркшоп 2](workshop-02-memory-normalization.md)

## Суть

Это практический воркшоп про первый слой AI-инфраструктуры продавца: как подключить источники данных, получить первые выгрузки или API-ответы и сохранить их так, чтобы дальше с ними мог работать агент.

Главный результат:

У участника появляется архитектура своей первой интеграции:

- что подключать;
- какие данные брать;
- куда складывать сырые данные;
- как не перезатирать историю;
- как агент будет этим пользоваться.

## Что показываем

### 1. Карта источников данных кабинета

Показываем, какие источники бывают:

- API маркетплейса;
- выгрузки из кабинета;
- рекламные отчеты;
- отчеты по продажам;
- остатки;
- поставки;
- карточки товаров;
- отзывы;
- Google Sheets;
- ручные файлы менеджера.

Практика:

Каждый участник составляет свою карту источников:

```text
Источник:
Какие данные дает:
Как часто нужны:
Как сейчас получаем:
Можно ли получить по API:
Где будем хранить:
Для каких проверок нужны:
```

### 3. Слой накопления: как не перезатирать данные и отчеты

Это важный блок. Если каждый запуск сохраняет файл с одним и тем же названием, история пропадает, а агенту не с чем сравнивать периоды.

Правило курса:

- `data/raw/` хранит исходники без изменений;
- `data/processed/` хранит обработанные версии;
- `data/normalized/` хранит приведенный к единому формату слой;
- `reports/` хранит выводы агента;
- `logs/` хранит историю запусков;
- `index.md` помогает быстро понять, какие данные уже есть.

Пример структуры накопления:

```text
data/
  raw/
    wildberries/
      sales/
        2026-07-03__sales__wb__raw.json
        2026-07-10__sales__wb__raw.json
      stocks/
        2026-07-03__stocks__wb__raw.json
    ozon/
      sales/
        2026-07-03__sales__ozon__raw.json
  processed/
    sales/
      2026-W27__sales__processed.csv
  normalized/
    products/
      products-current.csv
      products-2026-07-03.csv
reports/
  daily/
    2026-07-03__daily-check.md
  weekly/
    2026-W27__margin-check.md
  index.md
logs/
  data-runs.md
```

Правило именования:

```text
YYYY-MM-DD__source__entity__stage.ext
YYYY-WNN__check-name.md
```

Примеры:

```text
2026-07-03__wb__stocks__raw.json
2026-07-03__ozon__sales__raw.csv
2026-W27__weekly-margin-check.md
```

Что объясняем:

- сырые данные не редактируем руками;
- новые выгрузки сохраняем новым файлом;
- текущую актуальную версию можно держать отдельно как `*-current.csv`;
- каждый отчет должен иметь дату, период, источник и ссылку на входные файлы;
- если агент что-то пересчитал, он пишет это в лог.

Шаблон `reports/index.md`:

```text
# Индекс отчетов

| Дата | Период | Проверка | Источники | Файл отчета | Статус |
| --- | --- | --- | --- | --- | --- |
| 2026-07-03 | 2026-W27 | маржа | sales, costs, ads | weekly/2026-W27__margin-check.md | требует проверки |
```

Промпт:

```text
Создай правила накопления данных и отчетов для проекта.
Мне нужно, чтобы новые выгрузки и отчеты не перезатирали старые.

Опиши:
- структуру папок;
- правила именования файлов;
- что считается raw, processed и normalized;
- как вести reports/index.md;
- как агент должен записывать каждый запуск в logs/data-runs.md.
```

### 4. Получение данных из CSV / Excel

Показываем не анализ, а путь:

```text
сырой файл -> проверка структуры -> нормализация -> сохранение processed-файла -> отчет для AI
```

Что делает агент:

- смотрит структуру файла;
- определяет столбцы;
- предлагает нормальные названия;
- проверяет пропуски;
- сохраняет описание данных в Markdown.

Промпт:

```text
Посмотри файл из папки data/raw.
Не анализируй бизнес-результаты.
Сначала опиши структуру данных: какие столбцы есть, что они означают, какие есть проблемы, каких полей не хватает.
Предложи, как привести файл к нормальной структуре для дальнейшего анализа.
```

### 5. Получение данных по API

Показываем методологию:

1. Найти документацию API.
2. Понять, какие endpoints нужны.
3. Создать тестовый запрос.
4. Сохранить ответ в `data/raw`.
5. Нормализовать ответ в CSV / JSON.
6. Записать описание данных в Markdown.
7. Не хранить ключи в коде.

Важно:

Реальные ключи на занятии не светим. Используем тестовые данные или демонстрационный ключ.

Структура файла `.env.example`:

```text
WB_API_KEY=your_key_here
OZON_CLIENT_ID=your_client_id_here
OZON_API_KEY=your_api_key_here
```

Что объясняем:

- API-ключ не пишется в промпт;
- ключ хранится в `.env`;
- `.env` не отправляется в чат и не коммитится;
- агенту можно дать `.env.example`, но не реальные секреты.

### 6. MCP и локальные инструменты простым языком

MCP и локальные скрипты разбираем здесь, на первом воркшопе, потому что это часть подключения источников.

Объяснение без технической перегрузки:

```text
Если агенту нужно не просто читать файл, а ходить во внешний сервис,
ему нужен инструмент: локальный скрипт, API-клиент или MCP-сервер.
```

Примеры инструментов:

- получить список товаров;
- скачать продажи за период;
- скачать остатки;
- обновить Google Sheet;
- сохранить отчет;
- отправить Telegram-сводку.

На первом занятии мы не пишем сложный MCP-сервер. Мы проектируем карту:

```text
инструмент -> какие параметры принимает -> откуда берет данные -> куда сохраняет результат -> как агент его использует
```

### 7. Google Sheets как простой интерфейс

Объясняем:

Google Sheets может быть первым "интерфейсом" системы:

- туда можно складывать очищенные данные;
- менеджер может проверять руками;
- собственник может смотреть результат;
- AI может готовить структуру таблицы;
- позже можно подключить API.

Практика:

Проектируем таблицы:

```text
products
sales_raw
sales_processed
ad_costs
supplies
weekly_report
ai_findings
```

## Что делают участники

1. Создают карту источников данных.
2. Создают структуру проекта.
3. Выбирают первый источник: CSV, Excel, Google Sheet или API.
4. Описывают, какие данные нужны и зачем.
5. Создают `data-inventory.md`.
6. Создают `.env.example`.
7. Создают правила накопления: имена файлов, папки, `reports/index.md`, `logs/data-runs.md`.
8. Пишут первую задачу агенту на проверку структуры данных.
9. Проектируют первый локальный инструмент, API-клиент или MCP-команду на уровне схемы.

## Артефакты

- структура проекта;
- `data-inventory.md`;
- `.env.example`;
- описание первого источника данных;
- схема Google Sheets;
- правила накопления данных и отчетов;
- `reports/index.md`;
- `logs/data-runs.md`;
- карта первого инструмента или MCP-команды;
- понимание, как данные будут попадать в систему.

## Домашнее задание

1. Подготовить 1 реальную выгрузку или тестовый API-ответ.
2. Положить в `data/raw` по новым правилам именования.
3. Попросить агента описать структуру.
4. Создать `data/processed` версию или план нормализации.
5. Записать запуск в `logs/data-runs.md`.
6. Добавить отчет или описание файла в `reports/index.md`.
7. Не подключать реальные ключи без отдельной проверки безопасности.
