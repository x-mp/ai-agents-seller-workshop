# Воркшоп 2: системная память кабинета и нормализация данных

**Навигация:** [← README](../README.md) | [← Воркшоп 1](workshop-01-data-api-storage.md) | [Далее: воркшоп 3](workshop-03-cycles-skills-checks.md)

## Суть

Этот воркшоп про то, как создать и обслуживать постоянную память кабинета для AI-агента: товары, карточки, себестоимость, поставки, оферты, правила маржи, источники данных и связи между ними.

Главный результат:

У участника появляется управляемая память кабинета:

- файлы контекста;
- реестр памяти;
- граф связей;
- правила обновления;
- журнал изменений;
- первый нормализованный слой данных;
- понятный архив отчетов, который не перезатирает историю.

## Что показываем

### 1. Что такое системная память

AI-агент не должен каждый раз заново спрашивать:

- какие товары продаем;
- как считать маржу;
- какие комиссии учитывать;
- какие категории важны;
- какие оферты и условия действуют;
- какие поставки ожидаются;
- какие правила бренда соблюдать.

Все это нужно хранить в файлах или базе, но важно не просто "создать папку". Нужно понимать, кто обновляет эти данные, откуда они берутся, какие данные устарели и каким источникам можно доверять.

### 2. Папка `context/`

Пример:

```text
context/
  cabinet-context.md
  products.md
  product-costs.md
  marketplace-rules.md
  unit-economics-rules.md
  offers.md
  supplies.md
  brand-rules.md
  report-rules.md
  memory-index.md
  entity-graph.md
  source-of-truth.md
  update-rules.md
  memory-log.md
```

### 3. Инструменты управления памятью

Этот блок нужен, чтобы память не превратилась в свалку Markdown-файлов.

Что создаем:

```text
context/memory-index.md
context/entity-graph.md
context/source-of-truth.md
context/update-rules.md
context/memory-log.md
```

#### `memory-index.md`

Это оглавление памяти кабинета.

Пример:

```text
# Индекс памяти

| Файл | Что хранит | Источник | Как часто обновлять | Кто отвечает |
| --- | --- | --- | --- | --- |
| products.md | список товаров и SKU | кабинет / API | раз в неделю | менеджер |
| product-costs.md | себестоимость | внутренняя таблица | при изменении закупки | собственник |
| supplies.md | поставки | кабинет / файл поставок | перед поставкой | менеджер |
```

#### `entity-graph.md`

Это карта связей между сущностями.

Пример:

```text
# Граф кабинета

SKU-123
  -> карточка WB
  -> себестоимость
  -> цена
  -> остатки
  -> поставка 2026-07
  -> рекламная кампания
  -> отзывы
  -> задачи по визуалу
```

Зачем нужен граф:

- агент понимает, что товар связан не только с продажами;
- можно быстро увидеть, какие данные нужны для проверки;
- легче строить автоматизации и отчеты;
- проще объяснять разработчику, какие сущности связаны.

#### `source-of-truth.md`

Это файл, где зафиксировано, откуда брать правду.

Пример:

```text
# Источники правды

Себестоимость: внутренняя Google Sheet.
Остатки: API WB/Ozon.
Продажи: кабинет / API.
Рекламные расходы: рекламный кабинет.
Поставки: файл поставок + кабинет.
Карточки: кабинет маркетплейса.
Отзывы: кабинет / API / выгрузка.
```

#### `update-rules.md`

Правила обновления памяти.

Пример:

```text
# Правила обновления

- Себестоимость нельзя менять без подтверждения собственника.
- Остатки обновляются только из API или свежей выгрузки.
- Оферты обновляются вручную после изменения условий.
- Если данные старше 7 дней, агент должен пометить их как устаревшие.
- Если не хватает себестоимости, агент не делает вывод о марже.
```

#### `memory-log.md`

Журнал изменений.

Пример:

```text
# Журнал памяти

2026-07-03
- Обновили себестоимость SKU-123.
- Добавили новую поставку на 2026-07-15.
- Изменили правило минимальной маржи с 18% до 22%.
```

### 4. Команды агенту для работы с памятью

Промпт на создание памяти:

```text
Создай структуру памяти кабинета WB/Ozon.
Мне нужны файлы:
- cabinet-context.md;
- products.md;
- product-costs.md;
- supplies.md;
- offers.md;
- unit-economics-rules.md;
- memory-index.md;
- entity-graph.md;
- source-of-truth.md;
- update-rules.md;
- memory-log.md.

Для каждого файла объясни, что в нем хранить и как часто обновлять.
```

Промпт на проверку памяти:

```text
Проверь папку context/.
Найди:
- каких файлов памяти не хватает;
- какие данные устарели;
- где нет источника правды;
- какие связи между товарами, поставками, карточками и рекламой не описаны;
- какие данные нельзя использовать для выводов без обновления.

Сохрани отчет в reports/memory-audit.md.
```

Промпт на обновление памяти:

```text
На основе нового файла из data/raw обнови память кабинета.
Не перезаписывай важные правила без подтверждения.
Все изменения запиши в memory-log.md.
Если данные конфликтуют с source-of-truth.md, остановись и покажи конфликт.
```

### 5. Нормализация сырых данных

Показываем цепочку:

```text
data/raw/sales.csv
  -> data/processed/sales-normalized.csv
  -> reports/weekly/sales-summary.md
```

Задача агенту:

```text
Возьми сырой файл продаж.
Не делай финальный анализ.
Сначала приведи данные к нормальной структуре:
- понятные названия столбцов;
- единый формат дат;
- числовые поля;
- отдельные поля для SKU, названия, выручки, расходов;
- список проблем в данных.

Сохрани описание структуры в Markdown.
```

### 6. Накопление отчетов и данных после нормализации

На втором воркшопе закрепляем, что нормализация не должна жить отдельно от истории.

Задача:

- сохранить, какой сырой файл был использован;
- сохранить нормализованную версию;
- сохранить Markdown-описание;
- записать, какие выводы уже были сделаны;
- не потерять предыдущие отчеты.

Пример связки:

```text
data/raw/wildberries/sales/2026-07-03__wb__sales__raw.csv
  -> data/processed/sales/2026-07-03__wb__sales__processed.csv
  -> data/normalized/sales/sales-current.csv
  -> reports/data/2026-07-03__sales-structure.md
  -> logs/data-runs.md
```

Что добавляем в память:

- где лежат актуальные нормализованные данные;
- какие отчеты уже построены;
- какие данные устарели;
- какие источники нельзя использовать без обновления;
- какие регулярные проверки зависят от этих данных.

Промпт:

```text
Проверь слой данных и отчетов.

Найди:
- какие raw-файлы уже есть;
- какие processed и normalized версии созданы;
- какие отчеты ссылаются на эти данные;
- какие файлы могут перезатираться;
- где не хватает даты, периода или источника.

Сохрани вывод в reports/data/data-storage-audit.md.
Обнови reports/index.md и logs/data-runs.md.
```

### 7. Markdown как слой между таблицей и мышлением

Markdown удобен, потому что:

- его читает человек;
- его читает агент;
- его легко версионировать;
- в нем можно хранить выводы;
- он подходит для отчетов и инструкций.

Пример `sales-summary.md`:

```text
# Сводка продаж

Период:
Источник:
Количество строк:
Поля:
Проблемы данных:
Что можно анализировать:
Что нельзя анализировать без дополнительных данных:
```

## Что делают участники

1. Создают папку `context/`.
2. Заполняют `cabinet-context.md`.
3. Создают `products.md`.
4. Создают `unit-economics-rules.md`.
5. Создают `offers.md` или `supplies.md`, если это важно для бизнеса.
6. Создают `memory-index.md`, `entity-graph.md`, `source-of-truth.md`, `update-rules.md`, `memory-log.md`.
7. Берут один сырой файл и превращают его в Markdown-описание.
8. Формируют правило: какие данные агент должен читать перед анализом.

## Артефакты

- `cabinet-context.md`;
- `products.md`;
- `unit-economics-rules.md`;
- `offers.md` / `supplies.md`;
- `memory-index.md`;
- `entity-graph.md`;
- `source-of-truth.md`;
- `update-rules.md`;
- `memory-log.md`;
- первый Markdown-отчет о структуре данных;
- правила накопления нормализованных данных и отчетов;
- обновленный `reports/index.md`;
- понимание, как создавать, обновлять и проверять память кабинета.
